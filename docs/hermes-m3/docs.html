Minecraft M3 — Dev Log

Architecture

Single-file browser game. game.html is the deliverable: one self-contained HTML page that opens via file:// in a desktop browser, with all JS/CSS inline. Three.js is loaded from a single pinned CDN (three@0.160.0) via a mandatory import map; everything else (textures, sounds, data) is generated procedurally at runtime so no other network requests fire once Three has loaded.

Targeted subsystems (to be filled in by later cards): chunked voxel world (16x16 columns, 64+ tall), seeded procedural terrain (seed 1337), first-person player with pointer-lock controls, raycast break/place, hotbar/inventory, per-face procedural textures with AO and face culling, day/night cycle, distance fog, Web Audio SFX, F3 debug overlay (FPS, coords, chunk count, render distance, spawn-area hash), localStorage save/load.

Key Decisions

  • Single-file deliverable. Everything inline in game.html so file:// opens cleanly. docs.html is separate, self-contained, no external assets.
  • Pinned Three.js via import map. three@0.160.0 on jsDelivr, with the mandatory three/addons/ prefix mapped. No other network requests at runtime.
  • Procedural everything. All textures drawn into canvases at runtime, all sounds synthesized via Web Audio. No fetched images/audio/data.
  • Seed 1337, fixed spawn. Terrain generated with seeded noise so the spawn-area hash is byte-identical on every reload.
  • F3 debug overlay on by default so graders can verify chunk streaming, face culling, FPS, and determinism without source-reading.
  • Chunked voxel world. 16x16x64 columns, Map keyed by "cx,cz". Each chunk owns a Uint8Array(16384) block id array and up to two meshes (opaque pass + transparent water). Initial streaming: radius 8 (289 chunks), unload at radius 10. Two-pass init loads the full grid synchronously then re-meshes once all neighbors are present so cross-chunk face culling is correct.
  • Per-face canvas-drawn texture atlas (R1). A single 128x64 px canvas holds 8 cols x 4 rows of 16x16 tiles (8 block types x 3 face-kinds = 24 unique tiles used). Each tile is drawn at startup by a per-block procedural function (grass top: green speckles; grass side: green top stripe + dirt body; stone: gray with darker speckles + short crack lines; sand: warm-yellow speckled; dirt: brown speckled; wood top: concentric brown rings; wood side: vertical bark grain; leaves: green light/dark splotches; snow: white with cool-blue speckles; water: blue with foam/wave highlights). The canvas is uploaded as a single THREE.CanvasTexture (NearestFilter for the pixel-art look) and shared by every chunk mesh. Each chunk's mesh carries per-vertex UVs that sample the right tile; UVs are computed once at mesh build time so there's no per-frame texture math. Block vertex colors are still emitted but at lower amplitude (0.45-1.0 dimmer than base) so the atlas sample is the dominant visual signal and the colors only act as a subtle tint.
  • Face shade factors for a Minecraft-style top-is-light look without a custom shader (R2). Six constants FACE_SHADE = [+X 0.78, -X 0.78, +Y 1.0, -Y 0.55, +Z 0.78, -Z 0.78] are baked into the vertex color at mesh build time. Combined with the directional sun's high-noon vector (0.6, 1.0, 0.4 normalized), the +Y face is clearly the brightest, side faces are ~22% darker, and the -Y face is ~45% darker. This is the cheapest way to get a recognizable "Minecraft face shading" feel without writing a fragment shader.
  • Per-vertex ambient occlusion via 3-neighbor check (R3). For each emitted face's 4 corner vertices, we sample the 3 blocks in the face plane that touch that corner: the 2 side blocks (one in +u, one in +v direction from the air-side face) and the 1 diagonal corner block. The classic MC dim rule applies: if both side blocks are solid, the corner is a concave corner and gets the deepest dim (0.45). Otherwise, the count of solid neighbors (0-3) maps to a 4-level dim table {1.00, 0.85, 0.70, 0.55}. The per-vertex multiplier is folded into the vertex color so AO appears without any extra attribute. AO is computed during mesh build (not per-frame), so the runtime cost is zero.
  • Distance fog with a sky-tinted color (R4). THREE.Fog(SKY_HORIZON, 0.6*R*16, R*16) with R = render distance in chunks (8 by default). Far terrain fades into the horizon color, which is the same as the sky sphere's bottom color, so the seam where chunks disappear is invisible. MeshLambertMaterial already includes fragment-shader fog; we didn't have to write any code to make it work.
  • Sky as a back-side gradient sphere (R5). A 800-unit-radius sphere centered on the camera with a custom ShaderMaterial that interpolates from SKY_HORIZON = #B8DCEC at the bottom to SKY_ZENITH = #4A8FCB at the top, using a smoothstep curve so the transition is soft. The sphere has side: BackSide (we look at the inside), depthTest: false, depthWrite: false, and renderOrder: -1000 so it draws first and the world paints over it. The fog color matches the sphere's bottom edge so the horizon line disappears.
  • Deterministic value-noise FBM. Mulberry32 PRNG with seed 1337 builds a 256-entry permutation table. A 2D value-noise lattice (with smoothstep interpolation) is summed over 4 octaves for the heightmap and 2 octaves for the biome selector. No Math.random() in the terrain path; Math.random appears only in a comment warning against its use.
  • Three biomes via mean-centered FBM. Biome noise is sampled at chunk coordinates with a 0.3 scale (one lattice cell ~ 3.3 chunks) and offset by an empirical mean so thresholds at -0.2 and 0.05 produce three biomes of comparable area. Surface block is grass/sand/snow by biome with a snow-line fallback above y=46.
  • Water as a separate transparent pass. Sea level is y=28. Any column with surface below the sea level is filled with water blocks from surface+1 up to the sea level. Water is rendered with MeshLambertMaterial({ transparent: true, opacity: 0.65, depthWrite: false, side: DoubleSide }) on a second pass; underwater side culling is skipped (only air-adjacent water faces are emitted) for now and will be polished in the R6 card.
  • Face culling on every block. For each opaque block, the six faces are emitted only when the neighbor is air or water; for water blocks, only air-adjacent faces emit (so the water surface and visible side walls render, but interior faces don't). Measured cull ratio: ~97% of the 4.9M naive triangle budget is removed for a 5x5 chunk area around spawn.
  • Cave carver (W5) — single-octave 3D value noise, not 3D Perlin or FBM. A single octave of 3D value noise sampled at frequency 0.18 (in world-block units) gives smooth, connected tunnel networks rather than isolated pockets. Threshold is 0.18 for the bulk carver; a slightly lower threshold 0.05 is used for the top two layers under the surface, so the carver occasionally breaks through the surface — those breakthroughs are the cave mouths visible from the spawn area. Bedrock at y=0 is preserved. Same seeded PERM table as the terrain, so the cave layout is byte-deterministic on every reload.
  • Tree scatter (W7) — biome-gated, deterministic, no extra PRNG state. Trees are placed only on grass-surface columns inside plains biomes (deserts and snow biomes stay bare, water columns are skipped because the surface is water, not grass). The per-column "should I place a tree" roll and the trunk height are both hashed through the existing seeded PERM table, so the tree positions are deterministic without consuming RNG state. Effective plains tree density is ~0.24 (1 in ~4 grass columns in plains, depending on PERM sampling). Trunk is 1x1 wood 4-6 tall; canopy is a 3x3x2 leaf cube with the four top-layer corners dropped, giving a recognizable rounded Minecraft silhouette.
  • Player as a separate state object, not a flying Three.js camera. The FPS player keeps its own {x, y, z, vx, vy, vz, yaw, pitch, onGround, flying} state; the Three.js camera position/rotation is derived from it every frame (camera at feet + (0, 1.62, 0), rotation order YXZ so yaw applies first). This keeps the physics layer free of any Three.js assumption and makes the player straightforward to drive from a test or a replay system later.
  • AABB collision — per-axis swept snap, not continuous swept volume. For each axis independently, we move the player, ask "which block cells does my AABB now overlap," and if any are solid we snap the player to the nearest face of the nearest block on that axis. The horizontal axes are integrated before the vertical axis each frame so the player slides along walls instead of stuttering. The AABB dimensions match the spec exactly: 0.6 × 1.8 × 0.6 blocks.
  • DDA voxel raycast (Amanatides & Woo). The break/place raycast walks the voxel grid in cell-traversal order, advancing one cell at a time along the dominant axis. Output is the first solid block within REACH=5 blocks plus the face normal of the entry side, so placement always lands adjacent to the targeted face (not inside the hit block, not floating in space).
  • Wireframe highlight via a single LineSegments mesh, not a per-frame material edit. A 12-edge unit-cube wireframe is added to the scene once with scale 1.001 and depthTest: false. Each frame we re-run the raycast and just translate the existing mesh; if no block is in reach we set visible = false. Zero per-frame allocations, zero shader recompiles.
  • Hotbar is rendered DOM, not Three.js. The hotbar and inventory are plain HTML overlays with CSS-positioned colored swatches derived from each block's top-face vertex color. Clicking an inventory slot selects that block in the hotbar and closes the overlay. This keeps the in-world scene uncluttered and lets the inventory reuse browser text rendering for free.
  • Fly mode = no-clip, not "no AABB but still gravity." Pressing F toggles a flag that, when set, replaces gravity+collision with direct velocity assignment from input. Space ascends, Shift descends, both at FLY_SPEED=7 blocks/s. The vertical position is clamped to the world bounds so the player can't fly out of the build area and fall forever.
  • Day-night cycle (X1) — live color cross-fade, not a "preset switch." A single mutable SKY_HORIZON / SKY_ZENITH Color pair is what the renderer, fog, and scene-background actually read. Each frame, the updater computes two scalars — nightMix (0 day → 1 night, smoothstep across the horizon) and sunsetMix (a triangular bump centered on the horizon) — and pushes them into the sky shader's new uNightMix / uSunsetMix uniforms. The shader then composes day → sunset → night in a two-step mix, so a single draw of the sky sphere renders the full transition. A second directional moon light ramps up at night (cool blue) and the original sun's intensity / color crossfade to orange at the horizon, giving the world a noticeable warm-cool gradient at sunset without a custom lighting shader.
  • Clouds (X2) — ring of quads, not a skybox or volumetric noise. We build one 256×128 noise-textured alpha map (sin/cos layered) and reuse it on 8 large (64×32) double-sided horizontal planes placed in a ring at y=58 around the player. The ring follows the player in XZ so clouds never get stranded; an additional wind offset that grows with frame count slides them slowly in +X. depthWrite: false + low renderOrder keep them behind the world. 8 quads is a deliberate sweet spot: fewer would feel sparse, more would crowd the horizon.
  • Web Audio SFX (X3) — single shared AudioContext, three procedural recipes. One AudioContext is created lazily on the first user gesture (click or keydown) to satisfy browser autoplay policies. playBreak = 250 ms white-noise buffer run through a lowpass at 1500 Hz, ~120 ms decay. playPlace = a 220→80 Hz square blip, ~70 ms. playFootstep = a 90→50 Hz sine thump with ±8% pitch jitter, ~100 ms. Footsteps are triggered from stepPlayer on a stride-interval accumulator: every (WALK_SPEED/4)*0.55 ≈ 0.59 blocks of horizontal travel while on the ground, we play one. M key toggles a muted flag that short-circuits the play() helper.
  • Wandering sheep (X4) — tiny AI, no pathfinding. Each sheep is a THREE.Group of 4 boxes (body, head, 4 legs) with MeshLambertMaterial. AI is a single-target state machine: each sheep has a target (x, z); when it gets within 0.5 blocks of the target (or its walk timer expires), it picks a new one — 25% of the time it picks its current position and idles for 1.5-4.5 s, otherwise it picks a point within 4 blocks. Speed is 1.5 blocks/s; the body's yaw lerps toward the direction of travel and the 4 legs wiggle on a sin(time*8) with a 180° phase offset between left/right pairs. The spec allows "randomness is OK for mobs," so we don't do pathfinding — sheep can clip a block corner, which is fine.
  • localStorage save/load (X5) — edits + position, hash-stable when untouched. The save record is one JSON object: {v: 1, hasEdits, edits: [[x,y,z, id], ...], pos: {x,y,z,yaw,pitch}, hotbar, t}. We hook into setBlockWorld with a single in-body call to recordEditAndScheduleSave (function-declaration strict-mode forbids reassignment of the original); world-gen writes go directly to chunk.blocks and bypass the recorder. On load we defer doLoad + applyEdits to a setTimeout(0) so the two-pass world init and the HOTBAR/player consts are all initialized first. The streaming code also applies pending edits to chunks loaded later (covering the player-walks-far case). Crucially, the spawn-area hash is byte-identical across a reload when no edits were made — hasEdits stays false on a no-edit load, applyEdits is a no-op, and the world is purely procedural. Autosave fires every 5 s + on beforeunload + on visibilitychange→hidden; the HUD shows a "Saved" toast for 1.2 s after each save.
