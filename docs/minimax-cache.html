Does MiniMax-M3 prompt caching work?

A widely-shared report claims M3's Anthropic-compatible prompt cache is broken — "phantom", "stale", "no benefit". We tested it two ways from a clean room: caching works, and cache reads bill ~5× cheaper. The one genuine defect is a telemetry gap, not a caching failure.

Verdict

MiniMax-M3 prompt caching works. A request that re-presents a cached prefix returns a large cache_read_input_tokens and only a handful of fresh input tokens, and those cache reads bill ~5.2× less coding-plan quota than uncached input (independently matching a prior ~5.4× measurement). The only real defect is reporting: M3 never populates cache_creation_input_tokens (it is 0 on every call, including the write), which breaks write-cost accounting — but does not mean "nothing is cached." The viral "caching is a phantom/stale server bug" claim is a misdiagnosis: it used a popular copy-pasted prompt whose prefix was already warm in MiniMax's cache.

How MiniMax-M3 caching is supposed to work

Per MiniMax's Anthropic-compatible caching docs, you mark a stable prefix with cache_control: {"type":"ephemeral"}. The first request writes that prefix to a server-side cache; later requests that begin with the identical prefix read it back at a discount.

PropertyValue
Endpointhttps://api.minimax.io/anthropic (the SDK appends /v1/messages), header anthropic-version: 2023-06-01
Triggercache_control: {"type":"ephemeral"} on the cached block; minimum 512 input tokens
Reported asusage.cache_read_input_tokens (cache hit) and usage.cache_creation_input_tokens (write)
Discountcache-read input billed well below standard input (docs ~5×; current pay-go shows a discounted tier preserving the ~5× ratio)

The M3 deviation, up front: M3 serves cache reads correctly but never reports cache_creation_input_tokens — it stays 0 even on the write. Every confusion below traces back to people reading that 0 as "nothing was cached."

The claim under test

A report circulating in the MiniMax community, posted as "definitive proof of the M3 caching bug":

# tester.py output (the report's evidence)
Call 1: input_new=83,  cache_read=3698, cache_creation=0   ← "3698 tokens from nowhere"
Call 2: input_new=99,  cache_read=3698, cache_creation=0   ← "same stale value"
Call 3: input_new=115, cache_read=3698, cache_creation=0
Call 4: input_new=131, cache_read=3698, cache_creation=0
Call 5: input_new=147, cache_read=3698, cache_creation=0

Claims:
 (a) cache_creation=0 on every call  → "nothing is ever written to cache"
 (b) cache_read=3698 persists         → "reading stale/wrong data from other sessions"
 (c) therefore                        → "no cache benefit, phantom hits, server-side bug"

All three reduce to one belief: that a non-zero cache_read with a zero cache_creation means the cache is fake. We test that belief directly.

Test 1 — Does it actually cache? (cold-start, unique nonce)

The fix for the ambiguity above is a unique per-run nonce in the cached prefix, so every run starts cold: call 1 must be a write, and any later cache_read can only be this run's own content. We send a ~4,000-token system block (nonce + filler), marked cache_control: ephemeral, then vary only the user message.

# live run — MiniMax-M3, fresh nonce'd prefix
call  input_new  cache_read  cache_create  status
 1       4087        114          0         write   # cold: prefix is NEW → tiny cache_read
 2        119       4082          0         HIT     # prefix now served from cache
 3        119       4082          0         HIT
 4        119       4082          0         HIT
Caching works, unambiguously. Call 1's cache_read=114 (not 4082) proves the prefix was not pre-cached — so it cannot be "stale cross-session data". Calls 2–4 then return cache_read=4082, exactly the size of this run's nonce'd prefix, while fresh input collapses to ~119. If caching were off, every call would show ~4,000 fresh input and cache_read=0.

The one real defect: M3 doesn't report cache writes

Notice cache_creation_input_tokens = 0 on every row above — including the write (call 1). On a spec-compliant Anthropic endpoint, call 1 would report cache_creation_input_tokens ≈ 4000. M3 instead reports a tiny constant cache_read=114 on the write and cache_creation=0 throughout.

This is a genuine bug — but a telemetry bug, not a caching bug. Consequences:

  • Write-cost accounting breaks: you cannot see what the initial cache write cost from the usage block.
  • It fools naive testers: any tool that prints cached? = (cache_creation > 0) shows "NO" on every row even while caching is working. Judge caching by cache_read, never by cache_creation.

Everything else — the read discount, the prefix matching — works.

Why the "phantom/stale" report was wrong

"cache_read=3698 from nowhere on call 1"

The report used the popular copy-pasted prompt "You are a helpful assistant. " × 600 (~3,698 tokens). Because many people run that exact string, MiniMax already had it cached server-side — so the reporter's call 1 was an immediate hit on a warm shared prefix, not "tokens from nowhere". 3698 is the size of their own prompt, tracking their own content. Add a unique nonce (as in Test 1) and call 1 reverts to a real cold write (cache_read ≈ 114).

"persists → stale / cross-session"

cache_read equalling the prefix size on every call is exactly what a working prefix cache does — the cached prefix is identical each call. Our nonce'd run shows the value equals our prefix and changes with it, so it is this-session, not leaked. "no benefit / phantom" is also false — see the measured quota discount below.

Test 2 — Is it actually cheaper? (A/B on real quota)

"It reports cache reads" is not the same as "those reads cost less." To close that gap we ran a controlled A/B against the live coding-plan quota meter (the general lane of /coding_plan/remains): an uncached batch (a unique ~100k-token prefix per call → all fresh input) versus a cached batch (one warm-up write, then same-prefix reads → cache_read), measuring how much quota each burned. The ratio is taken on the fine 5-hour interval meter while total spend is hard-capped at 3% of the weekly quota.

The two requests, side by side (real requests from the run above, prefix truncated)

Same endpoint, same cache_control marker, same ~100k-token prefix — the only difference is whether that prefix repeats byte-for-byte:

# UNCACHED — a UNIQUE nonce makes the prefix new every call → it can never match the cache
POST https://api.minimax.io/anthropic/v1/messages        (header: anthropic-version: 2023-06-01)
{
  "model": "MiniMax-M3", "max_tokens": 16,
  "system": [{
    "type": "text",
    "cache_control": { "type": "ephemeral" },
    "text": "[ab-uncached run=…-90522 req=3 1780648…321] You are a helpful, precise assistant. …(×12500 ≈ 100k tok)"
  }],                                ▲ unique nonce → prefix differs every call
  "messages": [{ "role": "user", "content": "reply uncached 3" }]
}
# → response usage:
{ "input_tokens": 100090, "cache_read_input_tokens": 114, "cache_creation_input_tokens": 0 }
# the whole ~100k prefix is billed as FRESH input
# CACHED — a STABLE prefix (same bytes every call) matches the cached copy
{
  "model": "MiniMax-M3", "max_tokens": 16,
  "system": [{
    "type": "text",
    "cache_control": { "type": "ephemeral" },
    "text": "[ab-cached run=…-90522] You are a helpful, precise assistant. …(×12500 ≈ 100k tok — IDENTICAL every call)"
  }],                                ▲ byte-identical every call
  "messages": [{ "role": "user", "content": "reply cached 7" }]
}
# → response usage:
{ "input_tokens": 109, "cache_read_input_tokens": 100082, "cache_creation_input_tokens": 0 }
# only 109 tokens billed fresh; ~100k served from cache (~5× cheaper)

That is the whole mechanism. Keep the prefix identical and ~100,000 input tokens collapse to 109 fresh + 100,082 cache_read; change a single byte (the nonce) and you pay full price for all ~100k. Note cache_creation_input_tokens is 0 in both — that field never tells you anything on M3.

Safety of the A/B harness. The benchmark was adversarially reviewed (an independent GPT-5.5 pass) and made overspend-proof before running: it reserves the worst-case next-request cost before every send (so the integer/lagging quota meter cannot overshoot), routes the warm-up write through the same guarded path, aborts on any cache miss (a TTL miss would corrupt the ratio), and counts real throughput (input + cache_read) against the cap. It stopped well under budget — ~1% of weekly quota.

What this means for token-cost comparisons

In agentic coding runs the context is re-sent every turn, so the headline "millions of input tokens" are dominated by repeated prefix — which is exactly what caching discounts. With caching active, the real cost of those millions is closer to 1/5th of the raw throughput. Two conditions must hold for a run to actually capture that:

  • The harness must send cache_control on a stable prefix (the system prompt / tool preamble).
  • The prefix must stay byte-identical across turns. Aggressive context compression or rewriting changes the prefix and silently invalidates the cache — turning every "input" token back into full price.

So when comparing token spend across agents, "input tokens" is not a flat unit: a cache-read input token can be ~5× cheaper than a fresh one. The benchmark's token breakdown shows output tokens are tiny and comparable across models; the large input/throughput figures are where caching — when actually engaged — changes the economics.

The accurate bug report

If you are filing this with MiniMax, the defensible framing is narrow and correct — not "caching is broken":

works  Prompt caching functions on MiniMax-M3 via the Anthropic-compatible endpoint, and cache-read input bills ~5× cheaper (measured on quota).

bug  M3 does not populate cache_creation_input_tokens — it is 0 on every response, including the cache-write call. This breaks write-cost accounting and causes false "caching is broken" reports because tools key off cache_creation. Request: populate cache_creation_input_tokens on the write, to match the documented Anthropic-compatible usage schema.

Reproduce it yourself

A self-contained cold-start verifier (the essential ~30 lines). It uses the official anthropic SDK, so anyone can re-run it:

# pip install anthropic ; export MINIMAX_API_KEY=...
import os, anthropic
c = anthropic.Anthropic(base_url="https://api.minimax.io/anthropic",
                        api_key=os.environ["MINIMAX_API_KEY"])
run = os.urandom(6).hex()                          # unique nonce → cold start
sys = f"[probe {run}] " + "You are a helpful, precise assistant. " * 500   # ~4k tok, >512
for i in range(4):
    u = c.messages.with_raw_response.create(
        model="MiniMax-M3", max_tokens=16,
        system=[{"type":"text","text":sys,"cache_control":{"type":"ephemeral"}}],
        messages=[{"role":"user","content":f"say token{i}"}]).parse().usage
    cr = getattr(u,"cache_read_input_tokens",0) or 0
    print(i, "input", u.input_tokens, "cache_read", cr,
          "->", "HIT" if cr > 2000 else "write/miss")
# call 0 = write (cache_read≈114); calls 1-3 = HIT (cache_read≈4000). cache_creation stays 0.

The full verifier (cold-start + the quota A/B billing benchmark, with the overspend-proof safety caps) lives in minimax-cache-test.py — run python3 minimax-cache-test.py for the cache check, or MODE=ab python3 minimax-cache-test.py for the A/B (it hard-caps spend at 3% of weekly quota).

Methods & caveats

  • Quantization. The quota meter is integer-percent; the ~5.2× ratio carries ±~1-percent resolution. Two runs (5.2× and 5.4×) and the docs' 5× pricing ratio agree, so the figure is robust to that noise.
  • Scope. Test 1 proves M3 reports cache reads consistent with hits; Test 2 proves those reads cost less quota. Together they cover both "does it cache" and "is it cheaper".
  • Cache TTL. MiniMax expires the prefix based on system load; rapid successive reads (≈ every few seconds) stayed warm with zero misses across 93+120 reads. A long idle gap can let the prefix expire (a later read becomes a write) — the A/B aborts rather than mis-measure if that happens.
  • Endpoint. Anthropic Messages API (/v1/messages, anthropic-version: 2023-06-01), not the OpenAI-compatible path. The OpenAI path reports the same thing as prompt_tokens_details.cached_tokens.
