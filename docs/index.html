Build-a-Minecraft-Clone: four frontier coding agents, one identical prompt

Each model was handed the identical spec and asked to build the closest possible Minecraft clone as one self-contained game.html (Three.js, runs from file://, zero external assets) plus living docs.html. Graded on 130 points — Quality (100) + Documentation (10) + Speed (20) — from source inspection and live in-browser play, adversarially reviewed twice (Claude + Codex), with the report itself reviewed by Codex.

Contestants: MiniMax-M3 (Pi) · MiniMax-2.7-highspeed (Pi) · GPT-5.5 (Codex) · Claude Opus-4.8 · run mode /goal + dynamic workflow · world seed 1337.
▶ Play the four builds live: Opus-4.8 GPT-5.5 MiniMax-M3 MiniMax-2.7-highspeed — click the canvas to capture the mouse
Scored under the revised “working-integration” rubric. After Codex’s adversarial review of the games and of this report, the rubric was tightened to credit working integration, not announced intent — a feature merely defined (textures, AO) or claimed-but-broken scores zero. Quality is split into five source-evaluable buckets (below). This markedly separates the field and removes the earlier overly-generous 100/100s (rubric score; real bugs exist).

Verdict

All four shipped running voxel worlds — but only three are actually playable. Under integration-grading, capability and playability split apart: Opus-4.8 leads on capability (96.5) and GPT-5.5 trails it (94.5), both fully playable. MiniMax-M3 out-points 2.7 on raw capability (65.0 vs 60.0) — richer world, real texture/AO wiring — yet its core build loop is broken: duplicated mousedown handlers make every left-click break two blocks and every right-click place two, and the save never restores your position, so it is not playable as a Minecraft clone. 2.7 is the opposite — flat and feature-thin, but it plays. So on the playability-gated headline, 2.7 ranks above M3, and M3 lands last.

Two axes, on purpose. Capability = raw rubric points (what the build contains, wired-up). Playable? = can a person actually move, break, place, and persist — the brief asked for a playable clone. M3 wins the capability comparison with 2.7 but fails the playability gate, so the headline ranks playable builds first. (Overall also folds in Docs + Speed; M3’s Speed is unmeasurable after a platform incident.) The benchmark measures model + harness + plugins, not the model alone.

Caveats & incidents (transparency)

MiniMax server-side image-moderation incident (error 1026) — hit MiniMax-M3 mid-run. M3 verifies by reading rendered screenshots; partway through, MiniMax’s API began rejecting all image inputs as “sensitive” (a server-side change that also hit other users that day). It jammed at 34/35 todos; we rescued the session by stripping the flagged images and finished text-only. Consequence: M3’s wall-clock (Speed) is not comparable, and it never got vision self-verification like Opus did. Its deliverable is unaffected.
GPT-5.5 needed one later bugfix. Core build ~30 min; a separate later commit “Fix start overlay click handling” (~3 h later) fixed a real start bug. Speed uses ~35 min active labor (see leaderboard footnote for the strict-timing alternative).
Git hygiene. M3’s agent never made its own repo, so it committed into the parent harness repo (9 commits after 12f1738) and force-added its files past .gitignore. Cleanup: reset the harness to 12f1738 and give M3 its own repo.

The two adversarial reviews

Two independent passes drove these scores: a Claude workflow playing each game live in a browser, and Codex reviewing source (Codex also ran its own CDP runtime + screenshot pass, so the Claude live-play is a second, independent runtime check). Overlapping findings are high-confidence; Codex’s static findings (2.7 textures-not-wired, M3 fly-not-no-clip + save-overwrite) are what pulled those two down under integration-grading.

Most systematic failure — manifest honesty & placement. M3 and Opus put <!DOCTYPE html> before the FEATURE MANIFEST (spec required it first); 2.7-highspeed has no compliant top-of-file HTML manifest at all (JS comments inside the module); and several entries marked rubric features “implemented” that the source shows missing, stubbed, or broken.
This report was then reviewed by Codex (verdict: yes-with-fixes), which proposed the working-integration rubric adopted here. The score sheet now reflects the updated leaderboard and the timing limits called out above.

Why the revised rubric?

The old rubric awarded near-full credit for declared features. The revised rubric requires working integration: a feature has to be wired into the running game, not merely named in code or described in a manifest.

That matters most for MiniMax-2.7-highspeed: textures were defined but not mapped to the mesh (vertexColors only); the AO helper exists but is unused; block placement is source-broken; and inventory equals hotbar. Speed also uses GPT-5.5’s 35 min active labor, not calendar delay. M3 Speed is n/a because timing is unverifiable after the MiniMax incident.
