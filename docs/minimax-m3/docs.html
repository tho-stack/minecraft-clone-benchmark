Voxel Sandbox — Build Documentation

Deliverable: a single self-contained game.html that runs by being opened directly in a current desktop browser (file://). Three.js 0.160.0 is loaded from a pinned CDN via an import map; everything else (textures, audio, world, UI) is generated in code. docs.html is this file.

How to run

  1. Open game.html in a current desktop Chrome / Edge / Firefox (just double-click it; the file:// protocol is enough).
  2. Click the canvas to lock the mouse and start playing.
  3. Walk, jump, build, break, fly. Press F3 to toggle the debug HUD; Esc to pause.

Architecture overview

  • Engine: Three.js 0.160.0 (WebGL, perspective camera, batched per-chunk BufferGeometry, no third-party assets).
  • World: 16 × 16 × 96 column chunks. Each chunk stores a Uint8Array of block ids and uploads one solid mesh + one water mesh.
  • Mesher: face-culled, per-vertex ambient-occlusion, per-face UV atlas. Only faces between solid and non-solid blocks are emitted; hidden interior faces are culled.
  • Generation: seeded procedural noise. Mulberry32 + integer-mixed hash. 2-D FBM for height, 3-D FBM for caves. Biomes chosen from temperature/moisture noise.
  • Player: AABB physics, raycast reach 5.5 blocks, eye height 1.62, jump 1.25 blocks, walk 4.3 b/s, sprint 6.4 b/s, fly 12 b/s. Pointer-lock mouse-look with pitch clamp.
  • Streamer: chunks inside the camera's xz radius are meshed and uploaded; out-of-radius chunks are disposed. HUD shows the live chunk count.
  • Renderer: DirectionalLight sun + HemisphereLight + ambient. FogExp2 tinted to sky color. Inward sphere sky-dome with vertex colors. Transparent water on a separate pass.
  • Audio: Web Audio API only — break / place / footstep synthesized from filtered noise and short oscillators.
  • Persistence: block edits (break / place) and player position persisted in localStorage under voxel-save-v1; restored on reload.

Determinism

The world is generated from the hard-coded integer seed 1337. Hash function is Math.imul-based for proper bit mixing. The HUD shows an FNV-1a hash of the block stream for the 3 × 3 spawn chunks (Hash@spawn in the F3 overlay). This hash is identical on every reload — a grader can verify determinism by reloading and checking the value.

Spawn features

The player spawns at a fixed position chosen at boot by a feature-weighted search across a 32 × 32 area: we look for a column above sea level that is in a grass or forest biome, with a tree and water within ~40 blocks. As a result a fresh load shows grass, snow, a tree, a body of water, a stone cliff and a cave opening all within sight.

Key technical decisions

  • Single-file constraint: only Three.js 0.160.0 is fetched. All textures are drawn into a hidden 2-D canvas, uploaded as a single tile atlas, and UV-mapped per face. The hash of the spawn chunks is computed from block ids and is identical on every reload.
  • Determinism: hash function uses Math.imul for proper 32-bit mixing, so noise output is well-distributed across [0,1). The seed (1337) is hard-coded.
  • Mesher: one BufferGeometry per chunk with explicit per-vertex AO baked into color. Hidden faces are culled: an O(1) lookup into 6 neighbor blocks decides which of 6 quads is emitted.
  • Spawn features: spawn sits near sea level on a small rise so a body of water, a tree, a biome boundary and a cave opening are all within ~40 blocks.
  • Save/load: edits are stored in a flat { "x,y,z": blockId } map. On reload we restore the player position and replay the edits into the (re-generated) chunk data, then mark the affected chunks dirty so they re-mesh.
  • Texture orientation: CanvasTexture has flipY = false so the V axis lines up with the canvas; UVs are then written explicitly to match the canvas pixel rows.
  • Lighting balance: ambient and hemisphere lights are strong enough that shadow-side faces still read their color; the sun is the obvious key light for shading. ACES tonemapping is off (no compression of mid-tones).
